# PP 核心研发工作流 SOP

> **生效时间**: 2026-06-20  
> **适用项目**: Mingli-Cosmos（数字命理推演引擎）及未来所有 PP 系列项目  
> **地位**: 项目宪法 — 违反即视为工程事故，需在 DONE.md 中记录复盘  

---

## 前言：为什么需要这份 SOP

在过去一个月的密集迭代中，我们以极高效率完成了从零到 v4.0.0 生产就绪的全链路交付：186 项测试零失败、Docker 容器化、AI 流式对话、OpenClaw 风格管理后台、VPS 生产部署。但高强度推进也暴露出几类典型工程问题：

| 问题类型 | 真实案例 | 后果 |
|:---|:---|:---|
| **擅自回退安全机制** | UI 翻新时误删 L3 护栏层代码 | 生产环境出现未过滤敏感输出 |
| **先写前端空壳** | API 未跑通就搭建完整管理后台页面 | Failed to fetch，返工重做鉴权 |
| **UI 敷衍交付** | 配置表单交付为裸 HTML 标签，无样式无说明 | 非技术人员完全无法使用 |
| **头痛医头** | 改模型配置只改前端下拉框，未打通后端 Config API | 切换模型不生效，二次返工 |

**本 SOP 将这些教训固化为四条铁律，未来每一次编码对话必须默认遵守。**

---

## 铁律一：架构先行，拒绝盲目动工

### 1.1 原则

> 收到需求后，第一步永远是**全局探查**，而非立刻写代码。

### 1.2 强制流程

```
用户需求
  │
  ▼
Step 1: 现状探查（必须并行读取）
  ├── 读相关代码文件（前后端 + 数据库 schema）
  ├── 读 docs/arch/ 架构约束
  ├── 读 docs/contracts/ API 契约
  └── 读 docs/tasks/TODO.md 当前状态
  │
  ▼
Step 2: 差距分析（口头输出）
  ├── 数据库：是否需要新表/新字段？
  ├── API：是否需要新路由/新中间件？
  ├── UI：涉及哪些页面/组件？
  └── 联动：三者之间的数据流是怎样的？
  │
  ▼
Step 3: 实施计划（输出到 TODO 列表）
  └── 按「后端 → API → 前端」顺序排列任务
  │
  ▼
Step 4: 逐项实施 + 逐项验证
```

### 1.3 真实案例对照

| 正确做法 | 错误做法 |
|:---|:---|
| 做"模型热切换"需求时，先读 `loadConfig()` 发现它只读 `config.json`，再设计 DB-first 策略，最后改 Admin UI | 直接在前端加一个下拉框，后端完全不改，切换不生效 |
| 做"知识字典"需求时，先设计 `app_configs` 表结构 + `KnowledgeProvider` 服务层 + 5min TTL 缓存，再写管理后台表单 | 直接写一个 JSON 文件硬编码，后期无法动态编辑 |

### 1.4 检查清单

- [ ] 是否已读取至少 3 个相关源文件？
- [ ] 是否已确认数据库 schema 是否需要变更？
- [ ] 是否已确认 API 路由是否需要新增/修改？
- [ ] 是否已确认与现有功能的联动关系？
- [ ] 任务列表是否按「后端 → API → 前端」排列？

## 铁律二：强制防御性编程，严禁功能回退

### 2.1 原则

> 任何重构或 UI 翻新，必须以保证现有核心资产不丢失为绝对前提。  
> **重构导致原功能丢失 = 重大工程事故。**

### 2.2 核心资产清单（不可触碰）

本项目以下模块属于**一级保护资产**，任何改动前必须确认其完整性：

| 资产 | 文件位置 | 保护级别 |
|:---|:---|:---|
| **L1/L2/L3 防幻觉护栏** | `src/server/lib/anti-hallucination.ts` | 🔴 最高 |
| **输出护栏（敏感词+免责声明）** | `src/server/lib/guardrail.ts` | 🔴 最高 |
| **核心排盘引擎** | `src/engine/`（封版黑盒） | 🔴 最高 |
| **引擎 API 契约** | `docs/contracts/engine-api.md` | 🔴 最高 |
| **SSE 流式对话** | `src/server/index.ts` + ChatPanel | 🟡 高 |
| **Admin 鉴权中间件** | `src/server/modules/auth/` | 🟡 高 |
| **186 项全量测试** | `tests/` | 🟡 高 |
| **数据库 Schema** | `src/server/db/schema.ts` | 🟡 高 |

### 2.3 变更前双重验证

```
变更前:
  1. git diff 确认变更范围
  2. 核对变更是否触及「核心资产清单」
  3. 如触及 → 先跑 npm test，确认全绿后再改

变更后:
  1. npm test（186 项必须全绿）
  2. 如涉及护栏 → 手动验证敏感词拦截是否生效
  3. 如涉及 Admin → 验证鉴权中间件是否正常
```

### 2.4 真实案例对照

| 正确做法 | 错误做法 |
|:---|:---|
| UI 翻新时，先确认 `guardrail.ts` 和 `anti-hallucination.ts` 的 import 链完整，再改组件 | UI 翻新时直接重写组件文件，误删了 `guardrail.ts` 的调用链，生产环境裸奔 |
| 改 admin 路由前，先确认 `auth/index.ts` 中间件逻辑，保证所有 `/api/admin/*` 路由都有 Token 校验 | 加了一个新 admin 路由但忘了挂中间件，未登录就能访问 |

### 2.5 检查清单

- [ ] 变更前是否已 `git diff` 确认范围？
- [ ] 是否确认未触及「核心资产清单」？
- [ ] 变更后是否 `npm test` 全绿？
- [ ] 如涉及护栏/鉴权，是否已手动验证？

## 铁律三：B 端工业级 UI 质感标准

### 3.1 原则

> 严禁交付裸 HTML 标签、无样式表单、全屏黑框。  
> 后台配置项必须具备自解释能力，非技术人员也能零门槛使用。

### 3.2 UI 交付标准

#### 容器约束
- 所有页面内容必须包裹在 `Card` 组件内（shadcn/ui）
- 不允许出现无边界的全屏裸标签流

#### 表单自解释原则
- 每个输入框必须有 `description` 或 `text-muted-foreground` 说明文字
- 说明文字必须用**中文**写清楚"这是什么、怎么填、填了有什么用"
- 示例：
  ```
  ✅ 正确：「模型名称 — 在 SiliconFlow 控制台获取的模型 ID，如 Qwen/Qwen3.5-122B-A10B」
  ❌ 错误：「Model Name」
  ```

#### 双轨制命名
- **底层 Key（数据库/API）**: 使用拼音或英文驼峰（如 `zhengYin`、`modelName`）
- **前端 Display Name**: 使用完整中文（如「正印」「模型名称」）
- 禁止在前端直接展示拼音 Key

#### 状态反馈
- 所有异步操作（保存/删除/切换）必须有 loading 状态和结果 toast
- API 调用失败必须有明确的错误提示，禁止静默失败

### 3.3 默认技术栈

| 层级 | 技术 | 强制程度 |
|:---|:---|:---|
| 组件库 | shadcn/ui | 强制 |
| 样式 | Tailwind CSS 4 | 强制 |
| 图标 | lucide-react | 强制 |
| 布局容器 | `<Card>` + `<CardHeader>` + `<CardContent>` | 强制 |
| 表单 | react-hook-form + zod | 推荐 |
| Toast | sonner 或 shadcn/ui toast | 强制 |

### 3.4 设计基调

本项目采用「宣纸为底 · 朱砂为印 · 墨迹为字」的新中式美学：

| 色值 | 用途 |
|:---|:---|
| `#FBF7F0` | 宣纸底色（主背景） |
| `#1C1914` | 墨色（正文/标题） |
| `#B83A2E` | 朱砂红（强调色/印章） |
| `#D8D2C8` | 淡墨细线（分隔） |

### 3.5 检查清单

- [ ] 页面是否有 Card 容器？
- [ ] 每个输入框是否有中文说明？
- [ ] Key 和 Display Name 是否分离（双轨制）？
- [ ] 异步操作是否有 loading/error/success 反馈？
- [ ] 是否符合宣纸/朱砂/墨色配色？

## 铁律四：闭环测试与热生效

### 4.1 原则

> 严禁交付处于 `Failed to fetch` 断联状态的页面。  
> 前后端分离时，必须自行打通鉴权 Token 等拦截器机制。

### 4.2 闭环验证流程

```
代码写完
  │
  ▼
Step 1: 编译验证
  └── npm run build 或 tsc --noEmit → 零错误
  │
  ▼
Step 2: 单元测试
  └── npm test → 186 项全绿
  │
  ▼
Step 3: 功能验证（必须手动/自动化）
  ├── API: curl 或浏览器直接调用，确认返回 200
  ├── Admin: 确认 Token 鉴权正常，CRUD 生效
  ├── 前端: 确认页面渲染正常，无 console 报错
  └── 联动: 改配置 → 刷新页面 → 确认配置已生效
  │
  ▼
Step 4: 回归验证
  └── 确认未破坏现有功能（护栏/鉴权/排盘/对话）
```

### 4.3 热生效设计原则

- Admin 面板的配置修改必须**立即生效**，不需要重启服务
- 实现方式：配置存入 SQLite → `loadConfig()` DB-first 读取 → 每次请求实时加载
- 反例：把配置写死在 `config.json` 或环境变量里，改完要重启

### 4.4 真实案例对照

| 正确做法 | 错误做法 |
|:---|:---|
| 写 Admin LLM 配置页时，先确认 `loadConfig()` 改为 DB-first 策略，再写前端表单。改完在 Admin 面板切换模型 → 刷新主页 → 对话验证模型已切换 | 写完 Admin 表单后发现 `loadConfig()` 只读 `config.json`，改了不生效，需要返工重写后端 |
| 写完新 API 路由后，用 curl 测试一遍，确认返回 200 + 数据正确，再写前端代码 | 写完 API 不测试，直接写前端。前端报 `Failed to fetch`，不知道是 API 挂了还是鉴权拦截 |

### 4.5 检查清单

- [ ] `npm run build` 零错误？
- [ ] `npm test` 全绿（186/186）？
- [ ] API 端点是否手动验证过（curl/浏览器）？
- [ ] 鉴权 Token 是否已打通？
- [ ] 配置改动是否支持热生效（不改代码/不重启）？
- [ ] 是否已验证未破坏现有功能？
